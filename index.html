Productivity Tools

Due to confidentiality agreements, I cannot discuss most of my work, but I will do my best to give a general overview nonetheless. One of my earliest projects faced significant memory constraints on both clients and server. In order to circumvent those limitations, I used lessons from game development to distribute hi-res image processing across clients before sending data off to the server for caching. From there, I sought out opportunities to get my colleagues to delegate their most tedious tasks to me so that I could automate them away. As time went on, I got the opportunity to create more full stack applications from scratch (using various frameworks, databases, and cloud services).

Over the years, I have continued to work my way through the entire stack, starting from the frontend, going through the backend, and even venturing into automated deployments. Personally, I have found TypeScript to be invaluable throughout this process (in order to help maintain large-scale JavaScript codebases) and have enjoyed the ability to reuse code between client and server when developing Node.js and Electron applications. On a non-technical level, I frequently use mind maps to break down large projects and visualize their dependencies. I even wrote a prototype called Flow to help digitize this process and dust off my old trigonometry and collision detection skills.

Web Design

Most of my web work has been done through consulting and the projects showcased here are no exception. In both cases, I was called in to work on existing code. In the case of the Chrome Extension, Smart Screen, I was tasked with updating the layout using my knowledge of CSS (which is showcased throughout this entire page as the decorative devices surrounding every screenshot are entirely made up of vanilla CSS). In terms of the FPA Foundation website, I made accessibility and User Experience (UX) tweaks in addition to content edits.

Aside from the aforementioned projects, I enjoy working on multilingual sites and relish any opportunity to put my interest in Human Computer Interaction (HCI) to use in order to improve UX (e.g., subtly subitizing table rows so that the human eye can more easily distinguish between items on a subconscious level, or adding in multi-column sort/filter functionality). When page performance becomes an issue, I ruthlessly cull div soup and unnecessary JavaScript until the problem is solved (often times using the critically underutilized option that is CSS in their stead).

Game Development

Like many in the industry, I initially set out to learn programming in order to make video games, but I quickly fell in love with making productivity tools. However, before I switched my focus, I managed to make countless prototypes and actually finish four games. Using techniques outlined in Stanford's Developing Mobile Apps with Web Technologies course on iTunesU, I was able to create cross-platform applications across a variety of desktop and mobile operating systems to achieve every developer's dream of "write once, run anywhere".

The two most successful of these projects were Eternal Champions and Bloque It. The first was a brief game about learning financial literacy and ethics from Norse deities (furthermore, it is an "award-winning" RPG in the most technical sense of the term due to it having won 1st place in a state-level competition). Meanwhile, Bloque It was released on multiple app stores (i.e., iOS and FirefoxOS). It taught me many lessons that I still use to this day (e.g., how to optimize code to run smoothly on low budget hardware and how to deploy global updates that gracefully convert application data to new schemas).

Many More

In accordance with the ABCs of Writing (Accuracy, Brevity, and Clarity)--not to mention contractual obligations--I have omitted many other projects, such as volunteer and/or ongoing AI-related ones. But since many of those would retread familiar ground and/or be premature, I'll briefly mention the lessons learned on one of my more eclectic endeavors: becoming an author. Throughout my quest, I learned that ePub files contain XHTML. But sadly, using HTML and CSS is less straightforward than it would first appear...

Unfortunately, eBook readers are nearly as antiquated as most publishers' websites. Navigating the latter is often an exercise in archeology while the former still don't support CSS3 after all this time. To complicate matters further, browser vendors have only recently begun to implement parts of the CSS Paged Media Spec. Fortunately, paged.js provides a polyfill and an invaluable print preview feature, which makes it possible to generate print-ready PDFs for physical editions. If you'd like more details on that process, or anything else mentioned throughout this page, please do not hesitate to reach out!
